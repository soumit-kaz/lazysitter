# AET on Codex

Codex has no native typed subagents, so AET maps each agent to a **separate `codex exec`
process**. That is not a workaround for its own sake ‚ÄĒ it is what makes the pipeline's core
guarantee hold: every agent runs in a fresh context, seeing only the inputs the orchestrator
hands it, so **no agent can grade its own homework.**

## What runs where

- **Orchestrator** = the Codex skill at `.codex/skills/aet/SKILL.md`. Codex discovers skills
  in `.codex/skills/` at startup; the installed `AGENTS.md` block also points at it. When you
  say "run AET on \<feature\>", Codex loads the skill and becomes the Tier-0 orchestrator.
- **Agents** = role prompts in `.codex/skills/aet/agents/*.md`. The orchestrator spawns each
  by running:

  ```bash
  ./run-agent.sh <agent-name> <inputs-file> <output-file>
  # or on Windows:
  pwsh run-agent.ps1 <agent-name> <inputs-file> <output-file>
  ```

  `run-agent.sh` composes the role prompt + the task inputs and launches:

  ```
  codex exec --sandbox <s> --ask-for-approval <a> --skip-git-repo-check \
             --output-last-message <out> [--model <slug>] -
  ```

## The design ‚Üí Codex mapping

| Design intent | Mechanism |
|---------------|-----------|
| Context isolation between agents | one `codex exec` per agent ‚ÄĒ separate process, separate context |
| Model tiers | `--model` resolved from `models.env` via each agent's `TIER` |
| Least privilege | `--sandbox read-only \| workspace-write` from each agent's `.meta` |
| Git mutation gated | `--ask-for-approval on-request` for release/rollback (unless auto-approved) |
| Red-team ‚Č† build lineage | the `DISTINCT_MODEL` flag routes red-team to `MODEL_HIGH_ALT` |
| Budget / kill switch / audit log | orchestrator logic in `SKILL.md`; kill file `.codex/aet/KILL` |

The per-agent `.meta` files and the `TIER`/`SANDBOX`/`APPROVAL`/`DISTINCT_MODEL` values are
generated by the installer from `core/roster.json` ‚ÄĒ don't hand-edit them; edit the roster
and run `aet update`.

## Configure before first run

**`.codex/skills/aet/models.env`** ‚ÄĒ map tiers to real Codex model slugs. Blank = use your
Codex default model for that tier.

```
MODEL_HIGH=""        # architect, security-expert, closing-loop-auditor, ...
MODEL_HIGH_ALT=""    # red-team ‚ÄĒ SET THIS to a model different from MODEL_HIGH
MODEL_MID=""         # implementers, most verifiers
MODEL_LOW=""         # triage, explorer, secrets-scanner, ...
```

> If `MODEL_HIGH_ALT` is blank the red-team falls back to `MODEL_HIGH`, which weakens the
> "no shared blind spots" guarantee. `aet doctor` warns when this is unset.

**`.codex/skills/aet/aet.config.json`**

```json
{ "budget": 400000, "autoApproveGit": false, "monitorWindowMinutes": 30 }
```

- `budget` ‚ÄĒ token ceiling; the orchestrator pauses before exceeding it.
- `autoApproveGit` ‚ÄĒ for fully headless auto-merge, set `true`; the orchestrator then exports
  `AET_AUTO_GIT=1`, which downgrades release/rollback approval from `on-request` to `never`
  (they still run confined to `workspace-write`).

## Sandbox & network notes

- `read-only` agents can still *run* commands (grep, inspect, run read-only tools) ‚ÄĒ they just
  can't write files or reach the network.
- `dependency-auditor` may need network access for CVE/registry lookups. If your Codex sandbox
  blocks network under `workspace-write`, enable it for that agent in your Codex config.
- Nothing here uses `--dangerously-bypass-approvals-and-sandbox` / `--yolo`. Keep it that way.
